# MVP рекомендаций: цели и метрики

## Цели релиза (1–2 дня)
- Запустить гибридные рекомендации для гостей и авторизованных пользователей.
- Обеспечить безопасный fallback: если персонализация недоступна, показывать популярные товары.
- Не ухудшить стабильность: рекомендации не должны ломать основные страницы.

## Метрики качества
- CTR блока рекомендаций (клики по карточкам / показы блока).
- Add-to-cart from recs (добавления из блока / показы блока).
- Coverage (доля запросов, где блок не пустой), целевое значение: 100% с fallback.
- Latency серверной рекомендации (p95), целевое значение: <= 150ms.

## Минимальная аналитика событий
- `impression`: когда блок рекомендаций отрисован пользователю.
- `add_to_cart`: когда товар добавлен в корзину из recommendation placement.
- Обязательные поля события: `placement`, `product_id`, `user_id/session_id`, `created_at`.

## Релизный чек-лист
- Фича-флаги на каждый placement.
- Кэширование рекомендаций с TTL.
- Unit-тесты на ранжирование и fallback.
- Интеграционная проверка: `/`, `/products/{id}`, `/cart`.

## Денормализованная таблица для ML
- Таблица: `ml_training_interactions`.
- Назначение: хранить компактные взаимодействия пользователей с товарами для обучения модели на implicit-feedback.
- Одна строка = одно действие пользователя (`view`, `add_to_cart`, `purchase`, `click_recommendation`, `remove_from_cart`).

### Ключевые поля
- Актор: `user_id` или `session_id` (минимум одно из двух).
- Продуктовые признаки: `product_id`, `category_id`, `product_popularity`.
- Событие: `event_type`, `implicit_weight`, `placement`, `source_product_id`.
- Время: `created_at`.

### Наполнение таблицы
- При показе рекомендации пишется `view`.
- При добавлении из блока рекомендаций пишется `add_to_cart`.
- При оформлении заказа пишется `purchase`.
- Для уже накопленных `recommendation_events` и заказов доступен backfill.

### API доступа
- `GET /api/ml-dataset/samples` — просмотр строк датасета с фильтром по `placement` и/или `event_type`.
- `GET /api/ml-dataset/stats` — объём датасета, распределение по типам событий и прогресс к 1000 строк.
- `POST /api/ml-dataset/backfill` — восстановление датасета из существующих событий рекомендаций и заказов.
